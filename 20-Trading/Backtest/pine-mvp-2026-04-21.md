## 🔍 Знахідки

### ✅ Підтверджено
1. **Core structure detection працює** — pivot + sweep + BOS генерує ~1 сигнал/2 дні на 15m
2. **Marginal positive edge** на чистому BOS+2R exit (без фільтрів): +0.08R expectancy
3. **MCP → Pine → TV pipeline стабільна** — можемо швидко ітерувати

### ❌ Проблеми виявлено

**P1. Limit retest entries майже не заповнюються**
- Після BOS ціна експандує далі без pullback до entry level
- У 30 бар valid window з ~22 потенційних setup'ів — **0 заповнено**
- Це важлива знахідка: reтест-гіпотеза (SMC classic entry) **не працює механічно** на 15m EURUSD

**P2. TP1/TP2/TP3 partial exits блокують subsequent signals**
- Поки runner відкритий, `position_size != 0` → нові BOS signals ігноруються
- 22 signals → лише 3 entries при multi-TP model
- Потребує `pyramiding=1` або ID-separated entries

**P3. Sample size insufficient**
- 22 трейди << 50 мінімум для статистичної значущості
- TV Free дає ~5000 бар на 15m = 50 днів
- Для 200+ трейдів на 15m потрібно ~1 рік даних → TV Premium або D/4H TF

**P4. Filters disabled — не знаємо чи додають edge**
- Session KZ filter — не протестовано (зменшило б кількість трейдів до майже 0 на тестовому sample)
- HTF EMA bias — не протестовано
- Inner FVG — не спрацювало жодного разу через P1

### ⚠️ Caveats
- Pine strategy() не імплементує всі SMC concepts (discretionary OB selection, DXY confluence, news filter)
- Slippage=1 mintick використано; spread/commission=0 (нереалістично)
- 10% equity sizing у тестах — не реалістично (справжній 1% risk не перевірено в production)

## 💻 Pine Script — версії

### V1: Debug market-on-BOS (working baseline — 22 трейди)

```pine
//@version=6
strategy("SMC Debug — Market Entry on BOS",
     overlay=true, initial_capital=10000,
     default_qty_type=strategy.percent_of_equity, default_qty_value=10,
     pyramiding=0, slippage=1)

piv_len = input.int(5, "Pivot len", 2, 20)
sweep_lb = input.int(30, "Sweep lookback", 5, 100)
sl_buf_pips = input.float(5.0, "SL buffer pips", 0, 50, step=0.5)
rr = input.float(2.0, "Fixed RR target", 0.5, step=0.1)

pip = syminfo.mintick * 10
buf = sl_buf_pips * pip

ph = ta.pivothigh(high, piv_len, piv_len)
pl = ta.pivotlow(low, piv_len, piv_len)
var float last_ph = na
var float last_pl = na
var int last_ph_bi = na
var int last_pl_bi = na
if not na(ph)
    last_ph := ph
    last_ph_bi := bar_index - piv_len
if not na(pl)
    last_pl := pl
    last_pl_bi := bar_index - piv_len

var bool bl_sw = false
var float bl_sw_low = na
if not na(last_pl) and (bar_index - last_pl_bi) <= sweep_lb
    if low < last_pl and close > last_pl
        bl_sw := true
        bl_sw_low := low

var bool br_sw = false
var float br_sw_high = na
if not na(last_ph) and (bar_index - last_ph_bi) <= sweep_lb
    if high > last_ph and close < last_ph
        br_sw := true
        br_sw_high := high

bl_bos = bl_sw and not na(last_ph) and close > last_ph
br_bos = br_sw and not na(last_pl) and close < last_pl

if bl_bos and strategy.position_size == 0
    sl_px = bl_sw_low - buf
    risk = close - sl_px
    tp_px = close + risk * rr
    strategy.entry("L", strategy.long)
    strategy.exit("XL", "L", stop=sl_px, limit=tp_px)
    bl_sw := false

if br_bos and strategy.position_size == 0
    sl_px = br_sw_high + buf
    risk = sl_px - close
    tp_px = close - risk * rr
    strategy.entry("S", strategy.short)
    strategy.exit("XS", "S", stop=sl_px, limit=tp_px)
    br_sw := false

plotshape(bl_bos, "BOS↑", shape.triangleup, location.belowbar, color.lime, size=size.small)
plotshape(br_bos, "BOS↓", shape.triangledown, location.abovebar, color.red, size=size.small)
```

## 🎯 Пропозиції покращень

### A. Виправити limit-fill проблему

**A1. Expand valid_bars до 100+**
- Збільшити вікно очікування retest з 30 до 100 бар (~1 day)
- Ризик: застарілий setup може спрацювати коли bias вже змінився
- Mitigation: додати "bias still valid" check перед entry

**A2. Multi-level limit ladder**
- Замість 1 limit: 3 lime orders (BOS level, 50% retracement, FVG proximal)
- Збільшує fill-rate з ~0% до ~60% (estimated)
- Cost: positions з різним RR

**A3. Hybrid: market + pullback limit**
- Entry 50%: market на BOS bar (гарантований fill)
- Entry 50%: limit на BOS level (filled if retest happens)
- Blended RR, гарантовано якісь entries

**A4. Timeout conversion to market**
- Якщо limit не заповнюється за N бар → market order (якщо setup все ще валідний)
- Проблема: entry стає гірший в тренді, але гарантовано fills

### B. Виправити TP1/TP2/TP3 blocking

**B1. Increase pyramiding**
- `pyramiding=2` → дозволяє одночасно runner попередньої угоди + нова entry
- Проблема: sizing risk (2x позиції одночасно)
- Mitigation: sizing пропорційний — 0.5% на runner, 1% на нову

**B2. Force-close runner on new signal**
- При новому BOS signal: `strategy.close("L")` → потім нова entry
- Не відповідає real trading plan (TP3 може залишитись)
- Але для backtest — simpler stats

**B3. Separate strategy IDs per TP**
- `strategy.entry("L_tp1", ...)`, `strategy.entry("L_runner", ...)` — окремі позиції з різними exits
- Складніше, але кожна позиція має свій TP lifecycle
- Дозволяє нову entry одразу після TP1

### C. Розширити dataset

**C1. Переключитися на D (daily) TF**
- TV Free дає 5+ років даних на D (~1250 бар)
- 200+ потенційних signals — статистично значуще
- Cost: втрачаємо intraday edge (KZ, session-specific patterns)

**C2. Тестувати на кількох парах**
- GBPUSD, XAUUSD, US100, GER40 — та сама логіка
- Якщо edge подібний → universal; якщо ні → pair-specific
- `batch_run` MCP tool може зробити це автоматом

**C3. TV Premium upgrade**
- ~$15-30/міс, дає 10+ років даних на всіх TF
- Для серйозного бектесту — необхідно

### D. Додати фільтри та порівняти

**D1. Session KZ filter** — увімкнути, порівняти WR з baseline
**D2. HTF EMA bias** — те саме
**D3. DXY confluence via `request.security`** — додатковий screenshot-proxy
**D4. Consolidation filter** — skip entries в low-vol ranges

### E. Реалістичні витрати

**E1. Spread в commission**
- EURUSD typical 0.5 pip spread × 2 ways = 1 pip per round-trip
- На 15m де середній trade 20-30 pips profit → 3-5% витрат від profit
- Додати `commission_type=strategy.commission.cash_per_order, commission_value=5`

**E2. Slippage realistic**
- Slippage=1 mintick = 0.1 pip — замало
- Реалістично: 1-2 pips на market, 0 на limit (fill або ні)

### F. Pine strategy() vs Manual replay — decision

Після цього MVP стало ясно:
- **Pine strategy() корисний для**: швидкі A/B тести фільтрів (bias on/off, session on/off), параметричної оптимізації (pivot_len, sweep_lb)
- **Pine strategy() НЕ замінює** manual replay для: discretionary OB selection, DXY confluence, news filter, psychology — ці компоненти стратегії не автоматизуються
- **Decision:** Pine = supporting tool, manual replay = primary validation

## 🚨 Критичні знахідки

### 1. На D TF алгоритмічний BOS НЕ має edge
- 3 пари незалежно → всі від -16% до -33%
- WR 22-31% при 1:2 RR дає expectancy від -0.3R до -0.05R
- З реалістичними costs (~0.1R витрат) → однозначно збитково

### 2. HTF bias filter НЕ рятує — просто зменшує exposure
- Sample скорочено з 308 до 183 (−40%)
- WR практично не змінився (24.35% → 24.59%)
- P&L покращився (−32% → −19%) пропорційно менше трейдів
- **Висновок:** filter = reduced-risk loss, не edge

### 3. Fade (інверсія) УБИЧЕЙ працює ще гірше
- 4.97% WR на D EURUSD
- Причина: narrow SL (за BOS bar high) стопається нормальною волатильністю до того як 1:2 TP досягається
- **Не** означає що BOS — continuation signal; означає що симетрія base/fade не існує через різні SL distances

### 4. Algorithmic SMC ≠ Discretionary SMC
- Pivot-based BOS spam-генерує сигнали в чопі/консолідації
- "Valid" OB/FVG у SMC вимагає contextual judgment (HTF align, liquidity pool quality, session timing)
- Pine не може replicate це без ML/cross-reference data

### 5. Wide SL + 2R target = ambitious
- Sweep-to-BOS distance часто 50-150 pips на D
- 2R target = 100-300 pips move ПІСЛЯ entry
- Такий move на D TF — багатотижневий trend, рідкість

## 🎯 Оновлені висновки (після V3)

**Pine-based automated SMC strategy на тестованих TF/парах:**
- ❌ D TF — **негативний edge** на всіх 3 парах
- ⚠️ 15m TF — marginal edge +0.08R (22 trades, малий sample)
- ❌ Inverted — катастрофічно через asymmetric SL

**Hypothesis (треба перевірити):**
- H1/H4 TF може бути sweet spot (між spam на 15m і rarity на D)
- Session filter на 15m може відфільтрувати chop
- ATR-based SL замість swing-based SL може вирівняти RR

**Fundamental insight:**
Algorithmic BOS/sweep detection **не є edge самостійно**. Edge у SMC — це **discretionary judgment** яких OB валідні, куди йде liquidity, як interpretувати context. Pine автоматизує механіку але не судження.

## 🔀 Нові пропозиції (після V3 провалу)

### Варіант A: Hypothesis refinement (Pine-continue)
1. Test H1 TF — між 15m (chop) і D (rare)
2. Replace swing SL з ATR-based SL (2× ATR) — стабільна RR
3. Add volatility filter (ATR > X) — skip dead markets
4. Test on intraday futures (NQ, ES) — можливо forex не кращий market

### Варіант B: Honest pivot до manual Bar Replay
- Прийняти що Pine дав нам **negative evidence** про algorithmic SMC
- Переключитись на manual replay де discretionary judgment працює
- 50 manual trades за 1 тиждень дадуть більше edge-info ніж 1000 Pine trades

### Варіант C: Alternative algorithmic strategies
- Mean-reversion on RSI oversold/overbought + session
- Opening Range Breakout (ORB) — документований edge на forex
- Trend-following with moving average crossover + ATR filter
- Ці strategies не SMC, але дають baseline проти якого SMC повинен перевищувати

### Моя рекомендація: Варіант B + C (паралельно)
- **Manual replay** для валідації SMC edge (1-2 тижні)
- **Simple ORB strategy в Pine** як baseline (~1 година)
- Порівняти: якщо manual SMC > ORB → SMC edge існує (discretionary)
- Якщо manual SMC ≈ ORB → SMC це rebranded ORB + extra work
