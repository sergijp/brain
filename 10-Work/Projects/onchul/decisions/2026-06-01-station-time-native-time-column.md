# ADR-2026-06-01 — Зберігати час станцій як нативний MySQL TIME

**Status:** accepted
**Date:** 2026-06-01
**Стейкхолдери:** sergij.p

## Контекст

У таблицях `route_station` та `trip_station` час відправлення (`time`) і прибуття (`time_arrival`) зберігались як **JSON-рядок** у `string`-колонці: `{"HH":"08","mm":"00"}` (cast `App\Casts\Json`). Повна дата події = `trip.date + trip_day днів + HH:mm`.

Проблеми:
- Не сортується/не фільтрується нативно — SQL змушений робити `JSON_EXTRACT(time, "$.HH")` (порівняння рядків, не часу). Таких raw-SQL було 5 у 3 файлах.
- Немає валідації на рівні БД (`{"HH":"99"}` пройде; реальні дані містили навіть числовий `HH` без padding — `{"HH":9}`).
- Дублювання розбору `time['HH'].':'.time['mm']` у ~10 місцях.

Розглядали 3 варіанти: нативний `TIME`, хвилини від півночі (`SMALLINT`), JSON у value-object.

## Рішення

Зберігати в БД **нативний MySQL `TIME`** (`HH:mm:ss`, default `08:00:00`).

Ключовий хід: інтерфейс `{HH, mm}` у PHP та контракт з фронтендом/API **залишити незмінним** через новий cast-клас `App\Casts\TimeOfDay` (БД `TIME` ↔ PHP `['HH'=>'09','mm'=>'45']`). Це дало мінімальний блас-радіус — фронтенд (`RouteStation.vue`), PreProcessors, Resources, аксесори і `TripsService` не чіпали.

Конвертація наявних даних — окрема **ідемпотентна** міграція з guard по `information_schema` `DATA_TYPE` (no-op на свіжій БД, де create-міграція вже зробила `TIME`), з `down()`.

## Альтернативи (відкинуті)

| Варіант | Чому відкинули |
|---------|----------------|
| Хвилини від півночі (`SMALLINT` 0–1439) | Найпростіша арифметика, але менш читабельно в БД; не дає переваги над `TIME` для цього кейсу |
| JSON у value-object | Не вирішує головну проблему — SQL-запити/сортування/валідація лишаються кривими |

## Наслідки

**Позитивні:**
- Сортування/фільтри за часом працюють нативно; raw-SQL спрощено до `TIMESTAMP(date, time)`.
- Валідація формату на рівні БД.
- Фронтенд і публічний API-контракт не змінились (cast-обгортка).

**Негативні / трейдофи:**
- Cast-обгортка `{HH,mm}` лишається технічним боргом (зручніше було б Carbon назовні), але це свідомий компроміс заради малого блас-радіусу.

**Що це блокує / unlocks:**
- Подальші можливі оптимізації запитів пошуку рейсів.

## Перегляд

Якщо знадобиться відображати/рахувати час як повноцінний datetime назовні — мігрувати cast на Carbon і відрефакторити споживачів `time['HH']`.
